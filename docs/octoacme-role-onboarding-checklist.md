## Purpose
Ensure new team members joining in any role are set up for success and integrated into OctoAcme project processes from day one.

## General Onboarding (All Roles)
- [ ] Introduce new team member to Project Manager and Product Manager
- [ ] Grant access to project board (e.g., GitHub Projects), repo, and relevant documentation
- [ ] Share and walk through key process docs in `docs/`
- [ ] Confirm communication channels (Slack/Teams, email lists, etc.) and add to relevant groups
- [ ] Schedule 1:1 with Project Manager to align on expectations and current project status
- [ ] Add to regular meeting cadences (standups, sprint planning, demos)
- [ ] Review Definition of Done and team working agreements

---

## Role-Specific Onboarding

### UX Designer
- [ ] Access to design tools (e.g., Figma) and project design files
- [ ] Introduced to Product Manager to review current roadmap and user research
- [ ] Briefed on existing design system and component library
- [ ] Added to design review meetings and sprint demos
- [ ] Walkthrough of current product with Product Manager or senior team member
- [ ] Review any outstanding usability feedback or user research findings

### DevOps Engineer
- [ ] Access to CI/CD pipelines, cloud infrastructure, and monitoring dashboards
- [ ] Introduced to Developers and Project Manager for release coordination
- [ ] Review existing deployment runbooks and infrastructure-as-code repositories
- [ ] Onboarded to on-call rotation (if applicable)
- [ ] Review current incident response and escalation procedures
- [ ] Walkthrough of release process with outgoing or senior DevOps Engineer

### Business Analyst
- [ ] Access to requirements documentation, project charter, and stakeholder list
- [ ] Introduced to Product Manager and Project Manager for alignment on project context
- [ ] Briefed on discovery and requirements gathering processes
- [ ] Attend a stakeholder meeting or alignment session during first sprint
- [ ] Review existing backlog and acceptance criteria with Product Manager
- [ ] Introduced to Quality Lead to understand how requirements feed into test planning

### Support Lead
- [ ] Access to support ticketing system and monitoring dashboards
- [ ] Introduced to Developers and Project Manager
- [ ] Review current open support tickets and known issues list
- [ ] Briefed on post-release feedback process and bug triage cadence
- [ ] Added to release announcement distribution list
- [ ] Attend next release briefing or deployment review

### Quality Lead
- [ ] Access to test management tools, CI/CD pipeline, and staging environments
- [ ] Introduced to Developers and Project Manager
- [ ] Review existing test strategy, test plans, and automation frameworks
- [ ] Attend sprint planning to understand QA scope and capacity requirements
- [ ] Introduced to Business Analyst to align on requirements and acceptance criteria
- [ ] Review Definition of Done to understand quality gates

---

## Checklist Ownership
The Project Manager is responsible for ensuring this checklist is completed for each new team member. The checklist should be stored in the project repo (e.g., `docs/`) and linked from the onboarding issue or task.
